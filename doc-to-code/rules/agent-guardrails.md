## Agent guardrails (ограничения для предсказуемых изменений)

Дата: 2026-05-06

Цель: зафиксировать явные рамки изменений, чтобы автоматизация не приводила к “импровизации” и неконтролируемому refactor drift.

Источник: `doc-to-code/specs/doc-to-code-automation-plan.md` (раздел 5).

---

### Разрешённые области изменений (текущее предложение по структуре репозитория)

Агент может менять/добавлять файлы в:

- `ProcessView/**` (код приложения)
- `docx/**` (требования, маппинг, аспекты, трассировка)
- `PS-Scripts/**` (вспомогательные скрипты, если фича требует)

Запрещено менять без отдельного явного требования:

- любые файлы вне перечисленных директорий (если появятся в репозитории позднее)

---

### Зависимости

Текущее состояние:

- единственная явная зависимость NuGet: `Microsoft.Data.Sqlite` (см. `ProcessView/ProcessView.csproj`)

Правило:

- добавление/обновление зависимостей допускается только если это явно указано в документе фичи (`Data changes` / `Public surface` / `Constraints`) и в фазе анализа это отражено как отдельный пункт плана.

---

### Рефакторинг

Правило:

- запрещён “рефакторинг для красоты” и перестройка архитектуры без явного требования в фиче.
- разрешены минимальные локальные правки, строго необходимые для реализации фичи и прохождения базовой проверки из `doc-to-code/rules/dev-commands.md`.

---

### Недостаток информации

Правило:

- если в требованиях не хватает данных, агент не должен выдумывать поведение/контракты.
- пробел фиксируется как:
  - вопрос/уточнение в документе фичи, или
  - отдельное изменение документации (в `doc-to-code/`) до внесения кода.
