# План: автоматический перевод документации в код (предсказуемо, без CI)

Дата: 2026-05-06

Цель: настроить процесс, при котором агент может **детерминированно** превращать документацию (`docx`) в изменения кода, не “импровизируя”, и оставлять после себя актуальную трассировку “требование → реализация”.

---

## 1) Единый формат требований (шаблон фичи)

Нужен **стандартный шаблон** для каждой новой фичи (один `.md` на фичу). Без шаблона агент будет трактовать текст по‑разному.

Минимальные поля:

- `ID`: уникальный идентификатор фичи (например, `F-0007`).
- `Goal`: 1–2 предложения “зачем”.
- `Scope`: что входит / что не входит.
- `User flows`: 1–3 сценария (вход → действия → ожидаемый результат).
- `Data changes`: таблицы/поля/ключи (или “нет изменений в БД”).
- `Public surface`: какие команды/настройки/API меняются (или “не меняется”).
- `Acceptance criteria`: проверяемые пункты (без “улучшить/оптимизировать”).
- `Constraints`: что запрещено менять (контракты/схемы/форматы).
- `Implementation mapping`: *куда* в коде внедрять (файлы/классы/точки расширения).
- `Test plan (manual)`: как вручную проверить (команды/ожидаемые признаки).

---

## 2) Карта точек расширения проекта (implementation guide)

Нужен документ уровня “архитектурного оглавления”, чтобы агент понимал **куда добавлять** новое:

- структура проекта: проекты/папки и ответственность;
- основной цикл: точка входа, сбор данных, запись в SQLite;
- паттерн добавления новой фичи:
  - фича про сбор данных → где добавлять сборщик/interop;
  - фича про БД → где схема/репозитории;
  - фича про CLI → где парсинг аргументов/конфиг;
- правила именования классов/файлов;
- границы ответственности: что не трогать без отдельного требования.

Примечание: текущий `docx/implementation-mapping/process-attributes-current.md` полезен как инвентаризация, но это **не** общий гайд расширения.

---

## 3) Глоссарий и модель данных (термины и контракты)

Нужно зафиксировать словарь домена, чтобы агент не переосмысливал сущности:

- определения: `SnapshotId`, “снимок”, “процесс”, “сенсор”, “кластер нагрузки” и т.п.;
- стабильные ключи аналитики (например, `Name` vs PID) и почему;
- какие таблицы/поля считаются контрактом и не меняются без миграции.

---

## 4) Правила изменения схемы БД (даже без CI)

Если документация может требовать новые таблицы/поля, заранее фиксируются правила:

- можно ли менять существующие таблицы или только добавлять новые;
- как версионировать схему (минимум: таблица/поле “schema version”);
- принцип миграций (минимальный, ручной, без “магии”).

Без этого агент будет вносить изменения, которые сложно сопровождать.

---

## 5) Ограничения (guardrails) для агента

Чтобы получить предсказуемость, нужны явные рамки:

- какие директории/файлы разрешено менять;
- можно ли добавлять зависимости/пакеты (по умолчанию — нельзя без явного требования);
- можно ли проводить рефакторинг “для красоты” (обычно — нет);
- если информации не хватает — не выдумывать, а оформлять пробел как изменение документации/вопрос.

---

## 6) Протокол работы: двухфазная генерация

Чтобы избежать “сюрпризов”, внедряется протокол:

1. Фаза анализа (без изменений кода): агент читает документ фичи + карту расширения и формирует:
   - список затронутых файлов;
   - какие новые типы/таблицы/поля появятся;
   - как проверить вручную (test plan).
2. Фаза изменений: агент вносит правки **строго по утверждённому плану**.

---

## 7) Артефакты результата (что агент обязан обновлять)

После внедрения фичи агент обязан обновить:

- документ фичи (статус, итоги, ограничения);
- `docx/implementation-mapping/*`: трассировка “требование → где реализовано в коде”;
- (опционально) журнал изменений (`CHANGELOG.md` или отдельный лог).

Иначе документация быстро перестаёт соответствовать коду, и автоматизация деградирует.

---

## 8) Минимальный “инструментальный слой” (без CI)

Даже без CI нужен способ **локально** запускать одинаковые проверки:

- одна команда “собрать”;
- одна команда “базовая проверка” (например, запуск приложения с `--help` / быстрый сценарий).

Это может быть просто описано в документации; важно, чтобы агент всегда ссылался на один и тот же набор команд.
