## Скрипт процесса: подготовка документации фичи (agent playbook)

Назначение: пошаговое описание действий агента для подготовки “входных” требований к внедрению новой фичи так, чтобы реализация была максимально предсказуемой.

Это **не исполняемый скрипт**. Это договорённость “как агент работает с документами”.

---

### Входы

- Документ фичи в `doc-to-code/specs/features/<F-ID>.md` (или черновик от пользователя).
- Правила/гайд:
  - `doc-to-code/rules/agent-guardrails.md`
  - `doc-to-code/rules/implementation-guide.md`
  - `doc-to-code/rules/glossary-and-contracts.md`
  - `doc-to-code/rules/db-schema-and-migrations.md`
  - `doc-to-code/rules/dev-commands.md`
  - `doc-to-code/rules/artifacts-checklist.md`
- При необходимости: материалы из `docx/aspects/*` и `docx/implementation-mapping/*`.

---

### Выходы (обязательные артефакты)

- Документ фичи заполнен по шаблону (без TBD‑дырок, влияющих на код).
- В документе фичи есть:
  - явный план внедрения (файлы/точки интеграции),
  - проверяемые acceptance criteria,
  - test plan (manual) по стандартным командам.
- Если фича затрагивает схему/контракты:
  - в тексте фичи явно отмечено, какие контракты меняются/не меняются,
  - отражены последствия для миграций (если применимо).

---

### Шаги подготовки (фаза анализа, без изменений кода)

1) **Нормализация формата**
   - Привести документ фичи к структуре шаблона `doc-to-code/templates/features/F-XXXX-template.md`.
   - Убедиться, что есть уникальный `ID`.

2) **Проверка на полноту**
   - `Goal` не должен быть абстрактным (“улучшить/оптимизировать” без метрики).
   - `Scope` должен содержать non‑goals.
   - `User flows` должны быть проверяемыми.
   - `Acceptance criteria` должны быть булевыми (“выполнено/не выполнено”).

3) **Сведение с контрактами**
   - Сопоставить термины и сущности с `glossary-and-contracts.md`.
   - Если фича вводит новые термины — добавить их в документ фичи (раздел Notes/decisions) и запланировать обновление глоссария как отдельный шаг внедрения.

4) **Выбор точек расширения**
   - На основании `implementation-guide.md` перечислить:
     - какие файлы/классы затрагиваются,
     - где будет точка входа/интеграции.

5) **База данных и миграции (если применимо)**
   - Явно указать “Data changes”.
   - Если требуется миграция — описать минимальный подход (без реализации), и отразить риски.

6) **План проверки**
   - В `Test plan (manual)` использовать команды из `dev-commands.md`.
   - Добавить ожидаемые признаки в stdout/в БД/в файлах.

7) **Сверка с guardrails**
   - Проверить, что изменения укладываются в разрешённые директории и нет скрытых требований на зависимости/рефакторинг.

8) **Готовность к внедрению**
   - Выставить статус `Approved plan`, когда в документе нет “дырок”, критичных для кодинга.
