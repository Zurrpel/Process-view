## Скрипт процесса: пакет документации перед внедрением фичи

Назначение: описать минимальный набор документов, который должен быть готов *до начала кодинга*, чтобы агент мог внедрить фичу предсказуемо.

---

### Минимальный пакет (MVP)

1) **Спецификация фичи**
   - `doc-to-code/specs/features/<F-ID>.md`
   - заполнена по шаблону, статус: `Approved plan`

2) **Ограничения и правила**
   - `doc-to-code/rules/agent-guardrails.md`
   - `doc-to-code/rules/glossary-and-contracts.md`

3) **Точки расширения**
   - `doc-to-code/rules/implementation-guide.md`
   - в фиче заполнен раздел “Implementation mapping” с указанием файлов/классов

4) **Команды проверки**
   - `doc-to-code/rules/dev-commands.md`
   - в фиче есть test plan, который опирается на эти команды

---

### Дополнительно (если фича затрагивает БД)

- `doc-to-code/rules/db-schema-and-migrations.md`
- В документе фичи (`Data changes`) есть:
  - список таблиц/полей/ключей,
  - совместимость со старой схемой,
  - необходимость миграции (да/нет) и почему.

---

### Дополнительно (если фича влияет на аналитические документы)

- Обновления/ссылки на соответствующие документы в `docx/aspects/*`
- План обновления `docx/implementation-mapping/*` после внедрения (по `doc-to-code/rules/artifacts-checklist.md`)
