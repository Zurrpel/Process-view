## Карта родительских процессов: лучший подход

Этот документ описывает рекомендуемый, максимально быстрый и нативный способ построения карты родитель–потомок для всех процессов в Windows, а также его плюсы, ограничения и практические рекомендации по использованию в сценарии периодического опроса и записи в БД на основе **SQLite**.

---

### 1. Цель

- **Задача**: регулярно (с определённым интервалом) получать «снимок» всех процессов в системе с их родительскими PID и записывать эти данные в базу данных.
- **Приоритет**: максимальная скорость и минимальный оверхед, при этом данные должны быть достаточно точными для построения дерева процессов в момент съёма.

---

### 2. Краткий итог: что использовать

**Лучший вариант по скорости и нативности:**

- Использовать **WinAPI Toolhelp**:
  - `CreateToolhelp32Snapshot(TH32CS_SNAPPROCESS, 0)` — создаёт снимок процессов.
  - `Process32First` / `Process32Next` — перечисление процессов в снимке.
  - Структура `PROCESSENTRY32` даёт:
    - `th32ProcessID` — PID процесса.
    - `th32ParentProcessID` — PID родителя на момент создания процесса.

**Рекомендуемая реализация:**

- Небольшая утилита или сервис на **C#** / **C++**, либо PowerShell‑скрипт, который вызывает указанные WinAPI через `Add-Type`/PInvoke.
- Утилита возвращает структуру, удобную для записи в БД (например, JSON, CSV или прямую вставку в таблицу).

---

### 3. Почему не WMI/CIM (`Win32_Process`)

Класс `Win32_Process` (через `Get-CimInstance` / `Get-WmiObject`) действительно предоставляет `ParentProcessId`, но:

- **Производительность**:
  - Для полного дерева по всей системе WMI‑вариант может занимать **секунды** (6–15 секунд на больших системах), так как каждый запрос к WMI накладывает серьёзный оверхед (WMI Provider, COM/DCOM, marshaling и т.д.).
  - Toolhelp‑вариант (`CreateToolhelp32Snapshot`) в аналогичных сценариях показывает время порядка **сотен миллисекунд** на полный снимок.
- **Оверхед стек‑слоёв**:
  - WMI/CIM добавляет промежуточные службы, безопасность, преобразование объектов.
  - Toolhelp — это прямой WinAPI без дополнительных прослоек.
- **Вывод**:
  - Для **диагностических сценариев**, разовых запросов с богатой информацией (имя пользователя, командная строка, и т.п.) WMI/CIM удобно.
  - Для **частого, периодического опроса** всей системы WMI/CIM становится узким местом по производительности.

---

### 4. Детали подхода с Toolhelp (WinAPI)

#### 4.1. Основные функции

- `CreateToolhelp32Snapshot(TH32CS_SNAPPROCESS, 0)`:
  - Создаёт снимок всех процессов в системе на момент вызова.
  - Возвращает дескриптор снимка.
- `Process32First`:
  - Инициализирует перечисление процессов в снимке.
  - Заполняет структуру `PROCESSENTRY32` для первого процесса.
- `Process32Next`:
  - Последовательно получает следующие процессы из снимка.

#### 4.2. Структура `PROCESSENTRY32`

Ключевые поля, которые интересуют для построения дерева:

- `th32ProcessID` — PID процесса.
- `th32ParentProcessID` — PID родительского процесса на момент создания данного процесса.
- `szExeFile` — имя исполняемого файла (процесса).

Этого уже достаточно, чтобы:

- Построить карту `PID -> { ParentPID, Name }`.
- На основании неё построить дерево/лес процессов на момент снимка.

#### 4.3. Построение карты родителей

Общая логика:

1. Делаем снимок процессов (`CreateToolhelp32Snapshot`).
2. Проходим по всем элементам с помощью `Process32First` / `Process32Next`.
3. Для каждого процесса сохраняем:
   - `ProcessId = th32ProcessID`
   - `ParentProcessId = th32ParentProcessID`
   - `Name = szExeFile`
   - при необходимости другие поля.
4. Результат можно представить, например, как:
   - массив записей,
   - словарь `PID -> запись`,
   - или сразу сериализовать в формат, который удобно грузить в БД.

---

### 5. Ограничения и нюансы `th32ParentProcessID`

Важно понимать особенности поля `th32ParentProcessID`:

- Это **PID родителя на момент создания процесса**.
- Windows **переиспользует PID**:
  - Когда родительский процесс завершается, его PID может быть использован другим, несвязанным процессом.
  - Если смотреть только на текущее состояние системы через долгое время, можно ошибочно связать процесс‑«сироту» с новым процессом, получившим прежний PID.
- Чтобы гарантировать «идеальную» историю родитель–потомок во времени, система должна:
  - либо хранить дополнительную историю (время старта, события создания/завершения),
  - либо иметь открытый дескриптор на родительский процесс, чтобы не допускать переиспользования PID (обычно это не делается в прикладных сценариях).

**Вывод для нашей задачи:**

- Для **моментального снимка** (единичный опрос) данные `th32ParentProcessID` достаточно точны.
- Для **периодического опроса и записи в БД**:
  - Снимки с указанием времени (timestamp) позволяют корректно анализировать состояние дерева на каждый момент съёма.
  - Не стоит пытаться строить «вечное» дерево, опираясь только на PID без учёта времени и перезапусков.

---

### 6. Интеграция с базой данных SQLite

При периодическом опросе удобно сохранять данные в локальную файловую базу **SQLite** в таблицу (например, `ProcessSnapshot`) с такой минимальной схемой:

```sql
CREATE TABLE IF NOT EXISTS ProcessSnapshot (
    SnapshotId      INTEGER NOT NULL,
    ProcessId       INTEGER NOT NULL,
    ParentProcessId INTEGER NOT NULL,
    Name            TEXT    NOT NULL,
    PRIMARY KEY (SnapshotId, ProcessId)
);

CREATE INDEX IF NOT EXISTS IX_ProcessSnapshot_Snapshot_Parent
    ON ProcessSnapshot (SnapshotId, ParentProcessId);
```

Где:

- `SnapshotId` — целочисленный идентификатор конкретного опроса (может быть автоинкремент) **или** Unix‑время в секундах/миллисекундах.
- `ProcessId` (INTEGER).
- `ParentProcessId` (INTEGER).
- `Name` (TEXT).

При необходимости можно добавить дополнительные поля:

- `ExePath`, `CommandLine`, `StartTime`, `CpuTime`, `Memory` и т.п. (если вы дополнительно получаете их через другие API).

Рекомендации:

- **Минимизировать набор полей**:
  - Для начала достаточно (`SnapshotId`, `ProcessId`, `ParentProcessId`, `Name`).
  - Остальное можно добавить позже, если потребуется для анализа.
- **Использовать индексы SQLite**:
  - кластерный ключ по (`SnapshotId`, `ProcessId`) реализуется через составной `PRIMARY KEY`;
  - дополнительный индекс по (`SnapshotId`, `ParentProcessId`) даёт быстрые выборки детей для любого родителя в рамках одного снимка.

---

### 7. Сравнение подходов

| Подход                             | Скорость (примерно)         | Нативность | Сложность реализации | Наличие ParentPID |
|------------------------------------|-----------------------------|-----------|----------------------|-------------------|
| Toolhelp (`CreateToolhelp32Snapshot`) | Сотни миллисекунд на полный снимок | Высокая   | Средняя (нужен PInvoke/С++) | Да (`th32ParentProcessID`) |
| WMI/CIM (`Win32_Process`)         | Секунды на полный снимок    | Средняя   | Низкая (PowerShell/ .NET)   | Да (`ParentProcessId`)     |
| `Get-Process` / `System.Diagnostics.Process` | Миллисекунды на перечисление | Высокая   | Низкая               | Нет напрямую      |

**Приоритет для задачи построения карты родителей с высокой частотой опроса: Toolhelp.**

---

### 8. Рекомендации по архитектуре решения

1. **Вынести сбор данных в отдельный компонент (сервис/утилиту)**:
   - Этот компонент отвечает только за:
     - снятие снимка процессов через Toolhelp;
     - формирование набора записей `ProcessId/ParentProcessId/Name`;
     - передачу данных в БД (или в промежуточный файл/очередь).
2. **Стабилизировать формат данных**:
   - Описать фиксированный формат (JSON/CSV/табличный), чтобы аналитика и другие части системы могли опираться на него независимо от того, как именно реализован сборщик.
3. **Логирование и мониторинг**:
   - Логировать время выполнения каждого опроса (для контроля производительности).
   - При необходимости логировать случаи, когда снимок не удалось получить (ошибки WinAPI).
4. **Расширяемость**:
   - В будущем можно дополнить снимок дополнительными полями (CPU, память, командная строка), при этом ядро построения карты родителей останется тем же.

---

### 9. Вывод

Для вашей задачи — **регулярный опрос системы и построение карты родительских процессов с записью в SQLite‑БД** — наилучший баланс между скоростью, нативностью и контролем даёт использование:

- **WinAPI Toolhelp (`CreateToolhelp32Snapshot` + `Process32First/Next`, `PROCESSENTRY32.th32ParentProcessID`)** как источника данных;
- отдельного компонента (утилиты/сервиса или PowerShell‑скрипта с PInvoke), который:
  - быстро делает снимок,
  - строит карту родитель–потомок,
  - сохраняет её в компактном формате в БД.

WMI/CIM `Win32_Process` остаётся удобным, но для высокочастотного опроса и минимизации задержек он существенно уступает подходу с Toolhelp.

### 10. Оптимизация хранения и фильтрации снимков по времени в SQLite

#### 10.1. Базовая задача

Основной запрос к базе данных со снимками процессов — это **фильтрация по времени**:

- получить все процессы в заданном интервале времени;
- либо получить конкретный снимок (или несколько снимков) на момент \(T\), а затем отфильтровать их по другим критериям (PID, имя процесса, родительский PID и т.д.).

Чтобы такие запросы были быстрыми даже при большом количестве данных (десятки и сотни миллионов строк), важно:

- хранить время в явной отдельной колонке с индексом;
- минимизировать дублирование строковых данных;
- проектировать индексы под реальные запросы (по времени, по процессу во времени, по имени во времени).

#### 10.2. Разделение снимка и строк процессов

Текущая минимальная схема (см. раздел 6) хранит `SnapshotId` прямо в таблице `ProcessSnapshot`. Для эффективной фильтрации по времени удобнее выделить отдельную таблицу снимков:

```sql
CREATE TABLE Snapshot (
    Id        INTEGER PRIMARY KEY,   -- идентификатор снимка
    TakenAtMs INTEGER NOT NULL      -- Unix‑время в миллисекундах
);

CREATE INDEX IX_Snapshot_TakenAt ON Snapshot(TakenAtMs);
```

Таблица с процессами ссылается на снимок по `SnapshotId`:

```sql
CREATE TABLE ProcessSnapshot (
    SnapshotId      INTEGER NOT NULL,
    ProcessId       INTEGER NOT NULL,
    ParentProcessId INTEGER NOT NULL,
    Name            TEXT    NOT NULL,
    PRIMARY KEY (SnapshotId, ProcessId),
    FOREIGN KEY (SnapshotId) REFERENCES Snapshot(Id)
);
```

**Почему так быстрее для запросов по времени:**

- Время (`TakenAtMs`) вынесено в отдельную таблицу с **узкой, хорошо индексируемой колонкой**.
- Диапазонный запрос по времени (`WHERE TakenAtMs BETWEEN ...`) выполняется по индексу `IX_Snapshot_TakenAt` без сканирования всей таблицы `ProcessSnapshot`.
- После выбора нужных снимков по времени движок БД читает только связанные строки процессов, что уменьшает объём данных, проходящих через диск и кэш.

Примеры запросов:

- Снимки за интервал:

```sql
SELECT Id, TakenAtMs
FROM Snapshot
WHERE TakenAtMs BETWEEN $fromMs AND $toMs
ORDER BY TakenAtMs;
```

- Все процессы за интервал времени:

```sql
SELECT ps.*
FROM ProcessSnapshot ps
JOIN Snapshot s ON ps.SnapshotId = s.Id
WHERE s.TakenAtMs BETWEEN $fromMs AND $toMs;
```

#### 10.3. Нормализация имён процессов (замена имён на индексы)

При съёме снимка каждую секунду многие имена процессов (`Name`) повторяются из секунды в секунду (те же `svchost.exe`, `chrome.exe` и т.д.). Хранить строковое имя в каждой строке `ProcessSnapshot` неэффективно:

- расходуется лишнее дисковое пространство;
- индексы по `TEXT` занимают больше места и обрабатываются медленнее.

Поэтому целесообразно нормализовать имена:

```sql
CREATE TABLE ProcessName (
    Id   INTEGER PRIMARY KEY,
    Name TEXT NOT NULL UNIQUE
);
```

И заменить `Name TEXT` в `ProcessSnapshot` на `NameId INTEGER`:

```sql
CREATE TABLE ProcessSnapshot (
    SnapshotId      INTEGER NOT NULL,
    ProcessId       INTEGER NOT NULL,
    ParentProcessId INTEGER NOT NULL,
    NameId          INTEGER NOT NULL,
    PRIMARY KEY (SnapshotId, ProcessId),
    FOREIGN KEY (SnapshotId) REFERENCES Snapshot(Id),
    FOREIGN KEY (NameId)     REFERENCES ProcessName(Id)
);

CREATE INDEX IX_ProcessSnapshot_Name
    ON ProcessSnapshot (NameId);
```

**Эффекты нормализации:**

- Каждое уникальное имя хранится **один раз** в небольшой таблице `ProcessName`.
- Таблица `ProcessSnapshot` содержит только целочисленные идентификаторы имён — это уменьшает объём данных и размер индексов.
- Запросы вида «все появления процесса с именем X за период» становятся быстрыми:
  1. Поиск `NameId` в `ProcessName` по `Name`.
  2. Выбор строк из `ProcessSnapshot` по `(NameId, SnapshotId)` и временному интервалу.

Пример запроса по имени во времени:

```sql
SELECT s.TakenAtMs, ps.ProcessId, ps.ParentProcessId
FROM ProcessSnapshot ps
JOIN Snapshot s     ON ps.SnapshotId = s.Id
JOIN ProcessName pn ON ps.NameId    = pn.Id
WHERE pn.Name = $name
  AND s.TakenAtMs BETWEEN $fromMs AND $toMs
ORDER BY s.TakenAtMs;
```

#### 10.4. Индексы под типичные временные запросы

В зависимости от сценариев анализа можно добавить несколько целевых индексов:

- **По времени (уже показан):**

```sql
CREATE INDEX IX_Snapshot_TakenAt ON Snapshot(TakenAtMs);
```

- **По процессу во времени (история PID):**

```sql
CREATE INDEX IX_ProcessSnapshot_Process_Snapshot
    ON ProcessSnapshot (ProcessId, SnapshotId);
```

- **По имени во времени (история имени процесса):**

```sql
CREATE INDEX IX_ProcessSnapshot_Name_Snapshot
    ON ProcessSnapshot (NameId, SnapshotId);
```

При этом важно:

- не создавать лишние индексы «на всякий случай» — каждый индекс увеличивает размер БД и замедляет вставки;
- ориентироваться на реально выполняемые запросы (например, «снимки за время», «история конкретного PID», «все появления имени X за период»).

#### 10.5. Оптимизация записи для частых снимков

Если снимки делаются каждую секунду и каждый раз добавляется десятки или сотни строк, важно оптимизировать не только схему, но и режим записи:

- использовать **транзакцию на один снимок** (все INSERT‑ы за один `BEGIN … COMMIT`);
- по возможности вставлять несколько строк **одним `INSERT ... VALUES (...), (...), ...;`** вместо сотен отдельных команд;
- включить для базы:
  - `PRAGMA journal_mode = WAL;`
  - `PRAGMA synchronous = NORMAL;`

Такая комбинация (подтверждена практическими гайдами и бенчмарками SQLite) сильно ускоряет массовые вставки и делает фильтрацию по времени эффективной даже при больших объёмах данных.

#### 10.6. Шардинг по времени (при очень больших объёмах)

Если база растёт до десятков гигабайт, можно применить шардинг по времени:

- создавать отдельный файл БД на сутки/неделю (например, `process-snapshots-2026-03-13.db`);
- хранить в каждом файле таблицы `Snapshot`, `ProcessSnapshot`, `ProcessName` только за соответствующий период.

Преимущества:

- ограниченный размер каждой БД облегчает `VACUUM` и резервное копирование;
- запросы за короткий интервал времени работают только с одним файлом;
- архивные файлы можно переносить на более медленные/дешёвые носители.

Такой подход особенно полезен, если фильтрация по времени почти всегда выполняется в пределах последних часов или дней.

---

В совокупности:

- **явное хранение времени в отдельной таблице `Snapshot` с индексом по `TakenAtMs`;**
- **нормализация имён процессов через `ProcessName` и использование `NameId`;**
- **индексы, подобранные под реальные временные запросы;**
- **пакетные вставки в транзакциях + WAL‑режим;**

дают высокую эффективность как при длительной записи снимков, так и при последующей фильтрации данных по времени.
