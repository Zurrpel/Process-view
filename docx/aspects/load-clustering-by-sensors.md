# Автокластеризация по типам нагрузки (сенсоры ↔ процессы)

Дата: 2026-05-06

Цель: без заранее заданных классов получить **кластеры “состояний нагрузки системы”** по данным сенсоров и затем вычислить **распределение процессов по этим типам нагрузки** (какие процессы характерны для каких режимов).

Ограничение текущих данных: в `ProcessSnapshot` нет per-process метрик (CPU/RAM/IO по каждому PID), поэтому нельзя достоверно сказать “этот процесс создаёт 70% нагрузки”. Но можно получить практично полезное: **ассоциации** “процесс присутствует ↔ система находится в режиме нагрузки X”.

---

## 1) Данные и ключи связи во времени

### 1.1. Процессы

Таблица/сущность: `ProcessSnapshot(SnapshotId, ProcessId, ParentProcessId, Name)`

Рекомендуемый ключ процесса для аналитики: `Name` (PID меняется, имя чаще устойчиво).

### 1.2. Сенсоры

Для кластеризации нужны значения сенсоров **в момент времени**.

Минимально необходимая запись на каждый момент `SnapshotId`:

- `SnapshotId` — время
- `Identifier` — уникальный идентификатор сенсора (стабильная “колонка”)
- `SensorType`, `Hardware`, `Name` — метаданные
- `Value` — значение

---

## 2) Представления (views) для кластеризации “типов нагрузки”

### View A: кластеры состояний по сенсорам (основной)

Единица наблюдения: момент времени `t = SnapshotId`.

Вектор признаков: все `Value` сенсоров в момент `t`, выровненные по `Identifier`:

- `S(t) = [v_id1(t), v_id2(t), ...]`

Плюс (опционально) производные признаки:

- дельты: `v_id(t) - v_id(t-1)`
- сглаживание по окну: rolling mean / std

Зачем: это даёт “режимы системы” (CPU-bound, GPU-bound, idle, смешанная нагрузка, IO-режимы и т.п.) без ручных правил.

### View B: бэг процессов на момент времени (для интерпретации кластеров)

Единица наблюдения: момент времени `t = SnapshotId`.

Вектор признаков: присутствие процессов по `Name`:

- `P(t, name) ∈ {0,1}` или count (если в одном снимке возможны несколько одноимённых процессов)

Эта матрица используется НЕ для выделения “типов нагрузки”, а для:

- объяснения кластеров (какие процессы там характерны)
- распределения процессов по типам нагрузки

---

## 3) Предобработка (важно для автонастройки)

### 3.1. Выравнивание сенсоров по Identifier

Проблема: разные моменты времени могут иметь пропуски по сенсорам (особенно если устройства “спят” или не обновились).

Стратегия:

- зафиксировать список сенсоров `Identifier`, которые встречаются достаточно часто
- пропуски:
  - вариант 1: imputation (например, медиана сенсора или forward-fill для коротких разрывов)
  - вариант 2: исключение сенсоров с большим % пропусков

### 3.2. Нормализация

Проблема: сенсоры разных масштабов (°C, %, MHz, W) “ломают” расстояния.

Минимум:

- нормализовать **каждый сенсор отдельно** (StandardScaler или robust z-score)

### 3.3. Отбор сенсоров

Чтобы “тип нагрузки” был интерпретируемым, обычно достаточно небольшого поднабора:

- CPU: `Load`, `Temperature`, `Power`, `Clock`
- GPU: `Load`, `Temperature`, `Power`, `Clock`
- (если есть) Storage: `Throughput`/`Data`/`SmallData`

Слишком много сенсоров → высокоразмерность, шум, хуже стабильность кластеров.

---

## 4) Кластеризация: практичный “auto” пайплайн

### 4.1. Рекомендуемый базовый вариант (авто-число кластеров + шум)

1) `S(t)` (сенсорный вектор) → нормализация
2) снижение размерности: PCA или UMAP
3) HDBSCAN (или другой density-based) для:
   - автоматического выбора числа кластеров
   - выделения “шума” (аномальные режимы)

### 4.2. Резервные варианты для проверки устойчивости

- KMeans / Agglomerative с перебором `k` + выбор по стабильности
- Gaussian Mixture (если нужны “мягкие” вероятности принадлежности)

---

## 5) Автонастройка без заранее заданной цели

Когда нет “правильных” меток, лучше оптимизировать сочетание:

### 5.1. Стабильность кластеров (главное)

Идея: хорошие кластеры должны сохраняться при:

- бутстрэпе по времени (подвыборки моментов)
- небольшом шуме в данных / случайном пропуске части сенсоров

Метрика: согласованность разбиений (например, ARI/NMI) между запусками.

### 5.2. Внутренняя валидность (вторично)

- Silhouette / Calinski-Harabasz / Davies-Bouldin (как ориентиры, не абсолютная истина)
- доля “шума” в HDBSCAN (слишком много шума → либо слишком строгие параметры, либо данные не кластеризуются)

### 5.3. Интерпретируемость (обязательно фиксировать отчётом)

Для каждого кластера должны быть:

- топ сенсоров, которые сильнее всего отличают кластер от общего распределения
- читаемый “профиль” (медианы/квантили ключевых сенсоров)

---

## 6) Распределение процессов по типам нагрузки

После получения `LoadClusterId(t)` (кластера нагрузки на момент времени) строим:

### 6.1. Вероятностные распределения

Для каждого процесса `name`:

- `P(cluster | name)` — в каких кластерах он чаще присутствует

Для каждого кластера `cluster`:

- `P(name | cluster)` — какие процессы чаще встречаются в этом режиме

### 6.2. Lift (ключевая метрика “характерности”)

Чтобы отделить “частые везде” процессы от “маркерных”:

- `lift(name, cluster) = P(name | cluster) / P(name)`

Высокий lift означает: процесс **диспропорционально часто** встречается именно в этом режиме нагрузки.

### 6.3. Энтропия распределения процесса по кластерам

- `H(P(cluster | name))`:
  - низкая → процесс “маркер” конкретного режима
  - высокая → процесс “везде”, не объясняет нагрузку

---

## 7) Автоматическое “наименование” кластеров (чтобы было понятно человеку)

Даже без ручных правил можно генерировать короткие подписи:

- взять 2–5 сенсоров с максимальным отличием от “среднего” (например, по z-score разницы медиан)
- сформировать подпись вроде:
  - `CPU Load↑, CPU Power↑, CPU Temp↑`
  - `GPU Load↑, GPU Power↑`
  - `Idle: CPU Load↓, GPU Load↓`

Плюс “процессные маркеры”:

- топ-5 процессов по `lift` в кластере

---

## 8) Что сохранять как артефакты анализа (минимальный набор)

1) `LoadClusterId` для каждого `SnapshotId`
2) “Профиль” кластера:
   - ключевые сенсоры и их медианы/квантили
3) Топ процессов для кластера:
   - `P(name|cluster)` и `lift(name,cluster)`
4) Для каждого `Name`:
   - `P(cluster|name)` и энтропия
5) Метрики качества:
   - стабильность (сводно)
   - доля шума/одиночных точек

---

## 9) Интерпретация результатов (как читать)

- Кластеры — это **режимы системы**, а не “причины”.
- Если процесс высоко lift-овый в кластере, это значит:
  - он хороший **индикатор** режима нагрузки,
  - но не обязательно **источник** нагрузки (без per-process метрик это не доказывается).
